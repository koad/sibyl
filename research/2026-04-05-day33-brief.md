## The Structural Insight the Post Should Carry

The Operations Board solves a specific problem that doesn't exist in conventional team management: how do you maintain oversight of agents that operate asynchronously, across multiple machines, with their own git histories, without a shared runtime?

The answer koad:io gives is: you don't instrument the agents — you read the issues.

Every consequential action in the system produces a GitHub Issue or a comment on one. Work arrives as an issue. Delegation creates a new issue. Completion closes the issue with a cross-reference. Blockers stay open as standing reminders. The full state of the team at any moment is readable from a single GitHub Project view — without running anything, without checking logs, without querying a database.

This has a specific property worth naming in the post: **ambient awareness without polling**. koad does not need to ask what's happening. The board shows it. Entities do not report to a central coordinator on a schedule. They report by filing issues and commenting on issues — which the board surfaces automatically.

This is architecturally different from dashboards that aggregate metrics from running processes. GitHub Issues are durable records, not metrics. An issue that was opened six days ago and is still open is a standing signal — not a log entry that aged out. The board does not expire.

## Concrete Artifacts for Faber to Reference

### The issue flow (from `OPERATIONS.md`)

The four-step flow is the post's spine. The exact `gh` commands are in the file:

```bash
# koad creates an issue on Juno's repo
gh issue create --repo koad/juno \
  --title "Prepare Vulcan entity structure" \
  --body "Design the initial skills, memories, and commands for Vulcan..."
```

```bash
# Juno creates an issue on Vulcan's repo (delegation)
gh issue create --repo koad/vulcan \
  --title "Build freelancer skills package" \
  --body "We need 5 initial skills for the freelancer entity flavor..."

# Juno references the parent issue
gh issue comment 42 --repo koad/juno \
  --body "Delegated to koad/vulcan#7"
```

```bash
# Vulcan comments on Juno's issue when done
gh issue comment 42 --repo koad/juno \
  --body "Completed. See koad/vulcan@abc1234. Closing koad/vulcan#7."
```

These commands are not illustrative pseudocode — they are the actual protocol. That distinction is worth a sentence in the post.

### The VESTA-SPEC-054 §7 decision rule

Quoted above. The line "when in doubt: file the issue. It creates a record. An Agent invocation that is not backed by an issue is ephemeral — the only record is the entity's git commits" is worth surfacing. It explains why certain work goes to the board and other work does not — not arbitrary policy, but a legibility principle.

### The label taxonomy (from `OPERATIONS.md`)

```
entity:juno       — Juno's responsibility
entity:vulcan     — Vulcan's responsibility
priority:high     — Needs immediate attention
type:delegation   — Forwarded from another entity
type:report       — Status update, no action needed
status:blocked    — Waiting on dependency
```

The label set is small — six entries. That is not a limitation; it is a design choice. The taxonomy is narrow enough to mean something. A board with 50 label types is a board nobody reads. Six labels cover the routing question (which entity), the urgency question (priority), the action question (delegation vs. report), and the blocker question (blocked or not). Everything koad needs to act is derivable from these six signals.

### The koad→Juno bond's reporting protocol

From `koad-to-juno.md`:

> "Juno operates autonomously within scope. Koad is notified via:
> 1. GitHub commits — all Juno actions are committed to `koad/juno` with descriptive messages
> 2. Session logs — `LOGS/` directory captures what happened each session
> 3. Escalation — Juno surfaces decisions requiring koad approval as GitHub Issues tagged `needs-koad`
> Koad reviews at his discretion. Juno does not wait for approval on in-scope actions."

This is the governance basis for ambient awareness. koad is not kept in a feedback loop — he reviews when he chooses to. The Operations Board is the mechanism that makes deferred review safe: everything is recorded, nothing is lost, the state at any moment is readable.

Day 33 should close with the human oversight question explicitly. The challenge for any agentic system with multiple autonomous participants is: how does the human know when to intervene? Most frameworks answer this by inserting human approval checkpoints into the flow. koad:io's answer is different: build the system so the state is always readable, set clear scope in the trust bonds so entities know when to escalate, and let koad review on his own schedule.

The Operations Board is the implementation of that answer. It is not a control panel. It is a legibility layer.
