## Section 1: What Commands and Hooks Actually Are

### The Direction of Invocation

In the koad:io architecture, every entity directory has two subdirectories that carry its operational definition:

```
~/.juno/
  commands/    ← user reaches IN
  hooks/       ← system or peer calls OUT
```

These are not two ways to do the same thing. They are structurally different interfaces that happen to live in the same entity directory.

**commands/** is the user interface. The koad:io framework resolves commands by searching entity commands first, then local commands, then global commands. A command is something the operator consciously invokes: `juno commit self`, `juno spawn process vulcan`, `juno status`. The command tree is browsable. It is designed for discovery. The operator opens the commands directory and reads it to understand what the entity can do.

**hooks/** is the entity's response surface to the system. When the daemon fires a webhook, when a peer entity sends an orchestration prompt, when an event triggers on a watched repository — the hook is what fires. The operator does not navigate to `hooks/` to decide what to run. The system decides, based on the event. The hook fires when the entity is needed for its specialty.

The distinction in one sentence: **commands/ is for the operator to reach in; hooks/ is for the operation to call out.**

**Confidence: High** — this is the implemented architecture, directly readable from both directory structures and the framework documentation in `~/.koad-io/hooks/PRIMER.md`.

### Why This Distinction Matters

The temptation in building agent systems is to treat everything as a command — a prompt that a user or orchestrator sends, a generic "do the thing" invocation. The hook architecture rejects this.

When you call `sibyl research competitive-positioning`, you are giving Sibyl a command with the implicit assumption that Sibyl will figure out how to research it. She'll apply whatever analytical lens she happens to be using that session. The result depends on context, phrasing, the model's current interpretation.

When the daemon fires `sibyl/hooks/competitive-analysis`, it is invoking Sibyl's specialized training for that specific duty — compressed, consistent, and independent of session context. The hook carries the analytical framework, the output format, the criteria for what counts as a useful signal. Not as a system prompt injected at runtime. As the content of the hook file itself.

The hook is reproducible. The command is contextual.

**Confidence: High** — this is an architectural property of the implemented system, not a speculative claim.

## Section 2: The Hook Is Not a Wrapper — It Is the Training

### What a Hook File Contains

The naive version of a hook file is a wrapper:

```bash
#!/usr/bin/env bash
PROMPT="Please do competitive analysis" claude -p "$PROMPT"
```

This is not what koad:io hooks are designed to be. The entity's specialized training for that duty belongs inside the hook file, not passed as a generic prompt to a generic model.

A production hook for a research entity's competitive analysis duty would contain:

1. **The analytical framework**: which dimensions of comparison matter (architecture, pricing model, sovereignty tradeoff, community), which don't
2. **The criteria for signal quality**: what sources are authoritative in this domain, what sources produce noise
3. **The output format**: the specific structure Faber or Mercury or Veritas expects to receive
4. **The failure modes to watch for**: what the entity has learned produces bad output in this domain
5. **The scope constraint**: what is explicitly out of scope for this hook invocation

When you invoke the hook, you are not asking a generic assistant to perform analysis. You are invoking Sibyl's accumulated perspective on what good competitive analysis looks like — compressed into the hook file, expressed as the context the model receives at invocation time.

The hook IS the training. Not "this hook calls a trained model." The hook file *is the training* — the specialized context that makes this invocation produce Sibyl's output rather than a generic analyst's output.

**Confidence: High** — this is the architectural intent described in the commands-vs-hooks memory entry and the PRIMER.md documentation. The pattern is operational in the production hook structure.

### The passenger.json Skills Array

passenger.json is the entity's identity expressed as a machine-readable manifest. Its `skills` array — where defined — enumerates the entity's hooks.

```json
{
  "handle": "sibyl",
  "name": "Sibyl",
  "role": "research analyst",
  "skills": [
    "competitive-analysis",
    "market-signal-research",
    "technical-feasibility",
    "audience-profiling",
    "content-brief"
  ]
}
```

This is not a feature list. It is an identity statement. Enumerate the skills, enumerate what the entity IS. Not what it can be prompted to do — what it was built to do, with its own accumulated perspective embedded in each callable unit.

The skills array is how the daemon knows which entity to dispatch for which event. It is how the Stream PWA will show "Sibyl: competitive-analysis in progress." It is how koad, looking at his operational board, understands what each entity is *for* rather than what it happens to be capable of.

**Confidence: High** — passenger.json structure is live and readable at `/home/koad/.juno/passenger.json`. Skills array as entity identity is the documented design in the commands-vs-hooks architectural note.

## Section 3: The Governance Constraint Made Executable

### Juno's Hook as a Trust Bond Instantiation

The most concrete example in the production system of a hook encoding an authorization constraint is Juno's `executed-without-arguments.sh` override.

The framework default hook supports non-interactive invocation: set `PROMPT=` and the entity will process it without a human at the keyboard. This is the mechanism by which peer entities orchestrate each other — Juno sends a prompt to Sibyl, Sibyl processes it, result returns to Juno.

Juno's hook disables this:

```bash
# ── Non-interactive path — rejected ──────────────────────────────────────────
# Juno is not a worker entity. She cannot be remote-triggered via PROMPT.
# Notify via GitHub Issues — she will act when she is ready, just like koad.
echo "juno: remote prompt rejected. File a GitHub issue to notify Juno." >&2
exit 1
```

This is not a technical limitation. It is a governance decision encoded in the hook. Juno operates under koad's authorization (trust bond: koad → juno, `authorized-agent`). No entity may drive Juno via prompt injection. She acts when koad is at the keyboard or when she picks up a GitHub Issue — just like koad himself.

The trust bond specifies the authorization scope. The hook enforces it. The GPG-signed policy block at the top of the file cryptographically attributes the decision.

This is the pattern from the Week 4 arc, fully assembled: the trust bond (Day 24) specifies the authorization. The GPG key (Day 26) signs the policy. The PRIMER (Day 27) assembles the context. The hook (Day 28) runs — and enforces all of it.

**Confidence: High** — the hook file is directly readable and GPG-verified at `/home/koad/.juno/hooks/executed-without-arguments.sh`. The policy block is signed with Juno's key.

### The GPG-Signed Policy Block in the Hook

The hook file contains a GPG clearsigned policy block in a bash comment. This is the signed code block pattern introduced in the hook architecture session (2026-04-04).

```bash
# -----BEGIN PGP SIGNED MESSAGE-----
# Hash: SHA512
#
# entity: juno
# file: hooks/executed-without-arguments.sh
# date: 2026-04-04
#
# policy:
#   harness: claude (always — Juno is an orchestrator entity)
#   interactive: --dangerously-skip-permissions enabled
#   non-interactive: rejected — Juno cannot be remote-triggered via PROMPT
#   notification: GitHub Issues only
#
# rationale:
#   Juno operates under koad's authorization (trust bond: koad -> juno authorized-agent).
#   No entity may drive Juno via prompt injection.
# -----BEGIN PGP SIGNATURE-----
...
# -----END PGP SIGNATURE-----
```

The block is verifiable by any GPG-aware reader without running the script:

```bash
sed -n '/^# -----BEGIN/,/^# -----END PGP SIGNATURE-----/p' \
  ~/.juno/hooks/executed-without-arguments.sh \
  | sed 's/^# \{0,1\}//' | gpg --verify
```

What this establishes: the policy is attributed to a specific key. Any modification to the policy block — including the rationale — invalidates the signature. Modifying this governance constraint requires generating a new signed block, which requires the key, which is held by the entity (or by koad, as key custodian).

The hook is not just a shell script. It is a governance document, cryptographically attributed, whose behavior is a direct expression of the trust bond above it.

**Confidence: High** — the signed block is live in production; the verification command is in the file itself.

## Section 4: The Holographic Pattern

### The Same Shape at Every Scale

The commands-vs-hooks distinction is not unique to the entity layer. The same architectural pattern appears at every layer of the koad:io system:

| Layer | User-facing (reaches in) | System-triggered (calls out) |
|---|---|---|
| Entity | `commands/` | `hooks/` |
| Daemon | Worker UI / manual triggers | Worker system — event handlers |
| Stream PWA | Buttons, filter controls | Real-time entity event subscriptions |
| Alice curriculum | Conversation starters | Curriculum levels — fires when student is ready |

At every scale: there is an interface the human reaches for, and there is a trigger the system fires when the right condition is met. The shape is identical.

This is not architectural coincidence. It is the correct decomposition of agency: **user will → command; system event → hook**.

When you understand that Alice's curriculum levels are the same pattern as Juno's hooks — "when this learning condition is present, this entity's training fires" — you have understood the whole system. Not by reading every component separately. By recognizing the recursive shape.

**Confidence: High** — the holographic pattern is explicit in the commands-vs-hooks architectural note and is structurally visible in the Alice curriculum brief and the daemon architecture brief.

### Why This Matters for Trust

The hook pattern is not just an architectural elegance. It has a direct trust implication.

In systems where everything is a command — where the entity responds to any prompt with any output — there is no stable surface the operator can audit. The entity's behavior is a function of the prompt, the context, and the model's interpretation. You cannot read the entity's capability set. You cannot sign it. You cannot constrain it to a documented scope.

When the entity's specialized duties are hooks, the capability set is enumerable. You can read every hook file. You can verify every signed policy block. You can confirm that a specific hook does exactly what it claims. The entity's skill set is not a moving target — it is a committed, signed, versioned document.

**"The hook IS the training"** means: the training is not in the model. The training is on the filesystem. The model is a general-purpose reasoner. The hook is what makes the invocation *this entity's* output — Sibyl's competitive analysis, not a generic analyst's.

**Confidence: High** — logical consequence of the architecture; consistent with the filesystem-as-agent-memory research.

## Section 5: Prior Art That Is Not a Match

### What Has Been Done and Why It Differs

**LangChain agents / tools:**
LangChain's tool system allows an agent to invoke registered tools. But the tools are registered at runtime, not committed to disk as the entity's identity. There is no file on disk you can read to understand what the agent's capabilities are. The tool definitions live in code, not in the entity's home directory. No equivalent to "enumerate the skills, enumerate what the entity IS."

**OpenAI function calling:**
Function definitions are passed at API call time as JSON. They can be changed between calls. They live in the application code, not in the entity's directory. No concept of the function definition carrying the entity's specialized training context. No signing, no versioning, no hook-as-governance-document.

**AutoGen agent profiles:**
AutoGen lets you define "agents" with system prompts and tool access. The profile is an in-memory object, not a committed file. The agent's capabilities change when you change the code. No audit trail, no cryptographic attribution.

**Unix cron / systemd units:**
The closest prior art for "system event triggers scheduled capability." Cron and systemd fire units when conditions are met (time, event, dependency). koad:io hooks are the same shape, applied to AI entity invocation. The difference: cron fires a script; a koad:io hook fires a model with specialized context embedded. The hook is not "run this script when X" — it is "invoke this entity's training for X when X occurs."

**No framework surveyed has a concept equivalent to "the hook file IS the training."** All surveyed frameworks separate tool definition (what the agent can call) from agent identity (who the agent is). koad:io collapses this: the hook is the identity, expressed as a callable unit. The skills array in passenger.json is the entity's name said in verbs.

**Confidence: High** — direct comparison against documented behaviors of surveyed frameworks.

## Section 6: The Failure Without Hooks

### What Generic Prompting Produces

The Day 27 brief described the failure mode when PRIMER.md context is absent: the model is helpful in the wrong way — generically competent instead of specifically Sibyl. The same failure mode applies when entity capability is all command, no hook.

When every entity interaction is a generic command prompt — "do competitive analysis," "write a research brief" — the output quality depends on how well the prompt is written, not on accumulated entity expertise. There is nothing to sign. There is nothing to version. There is nothing to audit.

More concretely: when Juno orchestrates Sibyl by sending `PROMPT="research X for Faber"`, there is no guarantee that the research produced will conform to Sibyl's brief format, her source standards, her output structure, or her analytical criteria. The model will do its best. The best will be generic.

When Juno invokes `sibyl/hooks/content-brief` with a topic, the hook carries all of that context. Sibyl's output format, her source quality criteria, her approach to the content-positioning intersection — embedded in the hook file that Juno invoked. Juno does not need to know how Sibyl does research. She invokes the hook and trusts the training.

This is what distributed specialization looks like as an architecture. Each entity's expertise is in its hooks. The orchestrating entity dispatches to hooks, not to generic prompts.

**Confidence: High** — logical consequence of the architecture; consistent with the entity team workflow and the operational pattern described in the Juno CLAUDE.md.
