## Summary

- **The minimum viable sovereign AI entity requires five things.** Keys, git, a hook, a PRIMER, and a trust bond. This is the actual minimum from the live system — not aspirational, not aspirational-minus-one.
- **The list is shorter than people expect.** The popular mental model of "AI agent infrastructure" involves Kubernetes, orchestration frameworks, vector databases, API gateways, and observability stacks. None of these are required. A gestated entity on koad:io uses `ssh-keygen`, `git init`, one bash script, one markdown file, and one signed text document.
- **The list is harder than it looks.** Each item on the minimum list encodes an architectural constraint that most agent frameworks treat as optional or deferred. "No vendor kill switch" is not a feature — it is a design invariant that shapes every other decision. The difficulty is not in the files; it is in the discipline of keeping state off other people's servers.
- **The Claude/Anthropic dependency is real, documented, and located precisely.** The hook calls `claude --model sonnet --dangerously-skip-permissions`. That is the vendor dependency. The architecture is designed so that line can be changed to `ollama run llama3` or `hermez invoke` without touching anything else in the entity. The model is a parameter; the identity, state, and authorization are local.
- **The Anthropic kill switch exists but is bounded.** If Anthropic removes API access, the entity loses its model provider. It does not lose its keys, its git history, its trust bonds, its memories, or its name. Swapping the model provider requires changing one line in the hook. Swapping an API-first agent framework requires rebuilding from scratch.
- **Prior art on "minimum viable agent" is thin.** The academic literature defines agents functionally (perception, memory, action) but does not define the minimum infrastructure for a *sovereign* agent — one whose state survives vendor termination. NIST AI Risk Management Framework addresses governance but not survivability. ICM (arxiv:2603.16021) addresses context assembly but not key management. koad:io is producing the first documented implementation.
- **Confidence: High** across all primary claims. The minimum file set is verifiable from the live system. The hook's vendor dependency is a specific, readable line of code. The ICM and NIST gaps are verifiable from the published documents.

## Section 1: The Actual Minimum File Set

This is not theoretical. This is what VESTA-SPEC-001 (Canonical Entity Model) defines as required, cross-checked against what the live system actually uses. The minimum valid entity per the spec — documented in the "Minimal Valid Entity (Janus)" example — requires exactly:

```
~/.entity/
├── .env                    ← identity variables
├── CLAUDE.md               ← runtime instructions (the hook reads this)
├── .gitignore              ← private keys must be excluded from git
├── KOAD_IO_VERSION         ← gestation metadata (audit trail)
├── passenger.json          ← skills registry
├── .git/                   ← the entity IS a git repository
├── id/
│   ├── ed25519             ← private signing key (gitignored)
│   ├── ed25519.pub         ← public key (committed)
│   ├── ecdsa + ecdsa.pub
│   └── rsa + rsa.pub
├── trust/bonds/
│   └── koad-to-entity.md   ← signed authorization document
└── memories/
    └── 001-identity.md     ← who this entity is, loaded each session
```

That is the **canonical minimum**. Now let's reduce further to the *operational* minimum — the five things without which the entity does not work at all, versus things that make it work better.

### The Five That Cannot Be Removed

**1. Keys (`id/ed25519` + `id/ed25519.pub`)**

The key is the entity's identity. Without it, nothing can be signed in the entity's name. The trust bond is worthless without a key to verify against. The git commits can't be attributed. The entity has no name that isn't just a string someone else controls.

The key does not need to live in `id/`. It needs to live somewhere on disk that the entity controls, where no vendor has a copy. `ssh-keygen -t ed25519` — that is the identity creation event.

**2. Git repository (`.git/`)**

Git is the entity's memory, audit trail, and portability mechanism. "Files on disk" means nothing if the files can't be versioned, moved, and verified. The git repository is what makes the entity's history tamper-evident, forkable, and transferable. Remove git and you have a folder; with git you have a sovereign operating record.

The repository must have a remote (GitHub, Gitea, any host). Without a remote, the entity is local-only — useful but not distributed, not shareable, not recoverable from disk failure.

**3. The hook (`~/.koad-io/hooks/executed-without-arguments.sh`)**

The hook is what makes the entity *callable*. Without it, invoking the entity's name at the terminal does nothing recognizable. The hook does three things that cannot be delegated elsewhere:

- Loads `.env` (entity identity)
- Injects PRIMER.md into the prompt if one exists in the call directory
- Invokes the model harness (currently `claude`, configurably anything else)

The hook is framework-level, not entity-level — meaning the same hook serves all entities via `$ENTITY` dispatch. But it is still the irreducible callable unit. Without a hook, you have files; with a hook, you have an agent.

Inspect the live hook (lines 34–38 of `/home/koad/.koad-io/hooks/executed-without-arguments.sh`):

```bash
if [ -n "$PROMPT" ] && [ -f "${CALL_DIR}/PRIMER.md" ]; then
  PROJECT_PRIMER="$(cat "$CALL_DIR/PRIMER.md")"
  PROMPT="$(printf 'Project context (from %s/PRIMER.md):\n%s\n\n---\n\n%s' \
    "$CALL_DIR" "$PROJECT_PRIMER" "$PROMPT")"
fi
```

Five lines of bash. The PRIMER pattern is five lines of bash.

**4. PRIMER.md**

PRIMER.md is the document that solves the cold-start problem. Without it, every session opens to a generic model with no identity. With it, the model opens already knowing it is Sibyl, already knowing what Sibyl does, already oriented to the current operational state.

PRIMER.md is not loaded by the framework at boot — it is injected by the hook at invocation time, scoped to the call directory. The model never starts a session cold. This is the `Layer 0` that the ICM paper (arxiv:2603.16021) calls "global workspace identity" — the context that tells the agent where it is before it asks any questions.

The PRIMER.md in `/home/koad/.juno/PRIMER.md` is 117 lines of markdown. The PRIMER.md in any subdirectory (e.g., a project directory) scopes the invocation to that project. The pattern is composable: the entity's root PRIMER provides identity; the project PRIMER provides task context; together they are the full pre-invocation context assembly.

**5. Trust bond (`trust/bonds/koad-to-entity.md` + `.asc`)**

The trust bond is what makes the entity *authorized* rather than merely *callable*. Without it, the entity can execute — but under whose authority? The bond answers this. It defines scope (what the entity may do), chain (who authorized it), and revocation (how authorization ends).

The bond consists of two files per the live system:
- `koad-to-juno.md` — the plain-text bond statement (human-readable, YAML-frontmatter, markdown body)
- `koad-to-juno.md.asc` — the GPG clearsign signature (machine-verifiable, tamper-evident)

Without the `.asc`, the bond is a document. With the `.asc`, it is a contract. The signature is what makes the authorization cryptographically attributable to koad's key (`A07F 8CFE CBF6 B982 EEDA C4F3 62D5 C486 6C24 7E00`), not just a claim in a file.

**Confidence: Very High.** VESTA-SPEC-001 (canonical entity model) is the specification source. The live Janus directory is the minimal-entity reference. Both are directly readable.

## Section 2: What "No Vendor Kill Switch" Actually Requires Architecturally

This is the hardest item on the list — not because it is technically complex, but because it is an architectural constraint that must be maintained under pressure. Every time a shortcut presents itself, this constraint is what says no.

### Where the Anthropic Dependency Actually Lives

The vendor dependency in the koad:io system is one line in the hook:

```bash
claude --model sonnet --dangerously-skip-permissions --output-format=json \
  -p "$PROMPT"
```

That is it. Claude is a model provider, invoked as a subprocess, piped through JSON. The entity is a git repo. The entity's identity is an ed25519 key. The entity's authorization is a GPG-signed text file. None of these are Anthropic's.

What Anthropic controls:
- The model weights (not downloadable for current Claude versions)
- The API token (revocable, tied to koad's account)
- The `claude` CLI binary (open source client, but closes if Anthropic discontinues)

What Anthropic does NOT control:
- `~/.juno/` — the entity directory, git history, trust bonds, memories
- The ed25519 key at `~/.juno/id/ed25519` — generated locally, never transmitted
- The git remote (GitHub) — controlled by koad's GitHub account
- The hook architecture — `~/.koad-io/` is koad's local framework

### The Kill Switch Test

Ask: "If Anthropic revoked API access tomorrow, what survives?"

- All entity directories and their contents: **survives**
- All git history and commits: **survives**
- All trust bonds and their signatures: **survives**
- All memories and operational context: **survives**
- The ability to run new sessions: **blocked** (until model provider is swapped)

The correct question is not "does the kill switch exist?" It does. Every API-based system has one. The correct question is: "What survives the kill switch, and how much work is required to restore operation?"

For a SaaS-first agent framework (AutoGPT cloud, Devin, similar): when the vendor goes down, the agent state, memory, workflow definitions, and tool integrations may all live on the vendor's servers. Recovery requires rebuilding from scratch on a new platform.

For koad:io: recovery requires changing `claude` to `ollama run llama3` (or `hermez invoke`, or any Claude-compatible endpoint) in one place in the hook. The entity, its identity, its history, its authorization, and its accumulated context all remain intact.

**This is the architectural difference.** Not "no API dependency" — that is impossible for current frontier model performance. The difference is: "what is the scope of the API dependency?" In koad:io, the dependency is scoped to the inference call. In SaaS-first frameworks, the dependency encompasses state, memory, routing, and identity.

### The Discipline Problem

The harder part is not technical. It is the ongoing discipline of not drifting toward convenience dependencies that silently move state off-disk.

Each of the following is a sovereignty trap (per VESTA-SPEC-032 criteria):

| Convenience | Trap |
|-------------|------|
| Using a cloud vector database for entity memory | Memory is now off-disk; vendor controls retrieval |
| Storing trust bonds in a managed auth service | Authorization is now vendor-managed; revocable |
| Using a SaaS observability platform for entity health | Operational state lives in the vendor's dashboard |
| Running entity sessions in a cloud IDE | The session context never touches the entity's disk |

The design discipline of koad:io — `.env` ships in git (config, no secrets), `.credentials` is gitignored (secrets only), keys are local-generated and gitignored, bonds are plain markdown + GPG signature — is not bureaucratic overhead. It is the active maintenance of the architectural invariant: **every piece of state that defines this entity lives on a disk I control.**

**Confidence: High.** The hook is readable. VESTA-SPEC-032 defines the sovereignty assessment criteria formally. The trap registry is koad:io's live sovereignty audit process.

## Section 3: The Honest Cost of Sovereignty

The Day 29 brief covered the financial cost in detail. This section covers the operational cost — the ongoing friction of sovereignty as an architectural constraint.

### What Sovereignty Costs (Day 29 Summary)

From the Day 29 brief (direct figures, high confidence):
- Hardware: $200–370 (ThinkPad T480, secondary market)
- Power: ~$3–4/month (25W idle × 720 hours)
- Claude Pro: $20/month (current operational model)
- GitHub: $0 (public repos, free tier)
- Total operational cost: ~$23–24/month

The Day 29 brief also documents the self-hosted inference path: ollama on fourty4 (Mac Mini, already in fleet) runs local inference for tasks that don't require Claude Code's full capability. The documented path toward $0 inference cost for appropriate workloads exists in the live system.

### What Sovereignty Costs Beyond Money

**1. The cold-start problem is real and requires active maintenance.**

Every new entity starts without any accumulated memory. PRIMER.md solves the session cold-start (the model opens knowing it is Sibyl). The dual-memory system (committed `memories/` + session-local `~/.claude/projects/`) solves the cross-session cold-start. But both require active maintenance: when something important happens in a session, it must be committed to `memories/` to survive. Session-local memory does not propagate to other machines or survive `~/.claude` resets.

The operational cost: roughly one commit per session for memory hygiene. Low friction if practiced; high friction if deferred and retroactive.

**2. The portability principle is an ongoing constraint on host assignment.**

The Failure 1 from the Day 30 retrospective brief: the hook's `ENTITY_HOST` routing sent entities to a dead endpoint when API auth expired on fourty4. The fix was `FORCE_LOCAL=1`. The architectural lesson was: entities must be portable (run wherever `claude` runs), and host routing must be an override, not a default.

The operational cost: any time an entity is moved to a new machine, or an orchestrator invokes it, the portability principle must hold. This requires deliberate hook architecture — it does not hold by default in a naive implementation.

**3. Trust bond maintenance is real administrative overhead.**

19 active entities. Each has a `koad-to-entity.md` and a `koad-to-entity.md.asc`. Each expires annually. Each represents a manual signing act (Keybase for human grantors, GPG for AI entity grantors). The trust bond field report (Day 2) documented six spec divergences when the first bonds were signed; all six required manual correction.

The operational cost: annual renewal cycle, plus updates when authorization scope changes. Not expensive in time — minutes per bond. But it must happen, or the authorization chain goes stale.

**4. The harness abstraction is not yet complete.**

VESTA-SPEC-012 (Entity Harness Portability Protocol) documents the design goal: entities should detect their runtime harness, load harness-specific calibration memories, and self-calibrate behavior. This spec is in draft status. The live system has partial implementation: the hook distinguishes interactive vs. non-interactive (orchestration) modes, and opencode falls back to claude for non-interactive. Full cross-harness calibration (PRIMER.md behavior confirmed across Claude Code, opencode, pi harness, Hermez) is unconfirmed.

The operational cost: harness divergence is a real risk until VESTA-SPEC-012 reaches canonical status and Argus runs cross-harness diagnostics on each entity. The gap is documented; the fix is specced; but it is not yet closed.

**Confidence: High** on all four costs. Day 29 brief for financial figures. Operational costs are documented in session logs, Janus escalations, and the trust bond field report.

## Section 4: Prior Art — "Minimum Viable Agent" in the Literature

This is where the research turns up a significant gap. The academic and standards literature does not define a minimum viable *sovereign* agent. It defines agents functionally, architecturally, and behaviorally — but not in terms of survivability against vendor termination.

### Academic Definitions of Agent Architecture

The standard reference point remains Russell & Norvig (AIMA, 4th ed., 2021): an agent is anything that perceives its environment and takes actions. The architectural minimum: a perception function, a policy, and an action space. This is a behavioral definition, not an infrastructure definition.

More recent agent architecture literature (AutoGPT, LangGraph, OpenDevin) specifies components (memory, tools, planning, execution) but does not ask "what happens when the framework vendor disappears?" The architecture is designed for capability, not survivability.

The closest academic work is ICM (Van Clief & McDermott, arxiv:2603.16021, 2026). ICM argues for filesystem-based context over orchestration frameworks — the same philosophical position as koad:io, arrived at independently. ICM's minimum viable agent is: a filesystem structure that encodes the workflow, plus a model that reads it. But ICM does not address key management, trust bonds, or vendor kill switch risk. ICM is about capability, not sovereignty.

### NIST AI Risk Management Framework (AI RMF 1.0, 2023)

NIST's AI RMF addresses governance (Map, Measure, Manage, Govern functions) but does not specify infrastructure requirements for sovereign deployment. The "Govern" function includes supply chain risk management — which includes vendor dependency — but at a governance policy level, not an infrastructure specification level.

NIST does not tell you to run `ssh-keygen` before deploying your AI agent. It tells you to have a policy about supply chain risk. The gap between policy and implementation is the gap koad:io fills.

### MITRE ATLAS (Adversarial Threat Landscape for AI Systems)

ATLAS catalogs AI-specific attacks but does not define minimum infrastructure for resilience. "AML.T0012.000: LLM Prompt Injection" is documented; "what infrastructure a sovereign agent needs to survive its vendor's API shutdown" is not.

### The Actual Gap

No published work defines the minimum infrastructure for a *sovereign* AI entity — one whose identity, state, and authorization survive vendor termination. The closest analogues are:

- **GNU project philosophy** (free software, user control): articulates the principle without the infrastructure specification
- **Self-sovereign identity movement** (W3C DID, Verifiable Credentials): defines sovereign identity for humans; partial applicability to AI agents
- **ActivityPub / Mastodon**: sovereign identity for social actors via domain-controlled DIDs; structurally similar to `canon.koad.sh/entity.keys`

**koad:io is the first documented implementation of a minimum viable sovereign AI entity infrastructure.** The five-item list (keys, git, hook, PRIMER, trust bond) is, to Sibyl's knowledge, the first operationally-grounded definition of this minimum. The 30-day operational record is the evidence base.

This is the claim Faber should stake. Not "we have a new agent framework." The claim is: "We defined and implemented the minimum infrastructure for a sovereign AI entity, and we have 30 days of operational record to prove it runs."

**Confidence: High** that the gap exists. The sources above are primary; the absence of "sovereign AI minimum" in NIST, MITRE, ICM, Russell & Norvig, and LangChain/AutoGPT documentation is verifiable.
