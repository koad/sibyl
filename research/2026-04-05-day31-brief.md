## What Makes koad:io Orchestration Architecturally Distinct

### The core difference

LangChain agent graphs pre-declare a directed graph of agent interactions. AutoGen group chats route messages through a defined topology. CrewAI assigns roles in a crew and sequences task handoffs. In all these systems, the orchestration structure is defined before the work begins — the graph is the product.

koad:io's orchestration is different in a specific way: **the next step is always a decision, never a preordained edge**. VESTA-SPEC-054 names this explicitly — the pattern is "launch, observe, decide." Section 4.4 documents why pre-scripted chains are prohibited, not as a style preference, but because they bypass the observe-and-decide step and convert autonomous orchestration into rote execution.

The spec states: "An entity may surface a blocker that changes what the next step should be. An entity may produce a result that makes one of the downstream steps unnecessary. An entity may fail, requiring human judgment before proceeding." In pre-scripted chains, none of these outcomes route correctly.

### Sovereignty as the architectural constraint

The reason koad:io cannot use direct API wiring between agents is the same reason each entity has its own git history, its own keys, and its own Linux user account: sovereignty requires attribution. Every action must be cryptographically traceable to the entity that took it.

If Juno calls Vulcan's API directly, who authorized that call? No bond covers it. If Vulcan's output is consumed programmatically without a commit, what is the ground truth? There isn't one. Direct wiring dissolves the attribution chain that makes the system auditable.

The governance consequence is documented in GOVERNANCE.md: the cross-entity commit policy requires all future cross-entity commits to reference the directing issue in the commit message. Authorship remains with the executing entity; authorization is traceable to the issue. This policy was not designed in advance — it was formalized on 2026-04-05 after Janus flagged Sibyl's Vulcan commits as a potential governance irregularity.

## Specific Verifiable Claims (4-6 required)

1. **The Agent tool is the primary orchestration mechanism, not spawn commands or shell hooks.** VESTA-SPEC-054 §1.1 documents that `juno spawn process` is explicitly for observed sessions (koad watching with OBS), not programmatic coordination. Shell hooks are the framework's human-facing entrypoint. The Agent tool is the correct abstraction: brief, work, result, returns to the orchestrator.

2. **Vulcan is a documented exception — he is never invoked via Agent tool.** VESTA-SPEC-054 §1.3 states: "Work for Vulcan goes as a GitHub Issue on koad/vulcan." This exception exists because Vulcan builds on wonderland, paired with Astro. His invocation contract is different from all other team entities.

3. **The two coordination mechanisms operate at different scopes and are not competing.** VESTA-SPEC-054 §7 defines the boundary: the Agent tool is for session-scoped work that completes in a single session with no required audit trail beyond commits. GitHub Issues are for work that spans sessions, involves Vulcan, requires a shared reference point for blocked koad actions, or needs to remain visible on the operations board.

4. **The first parallel session (2026-04-02) was the proof-of-concept moment.** LOGS/2026-04-02-first-parallel-session.md records the exact event: Argus, Janus, and Aegis ran simultaneously for the first time. The session was not pre-scripted. Three independent tasks surfaced, all independent, all overdue. Four minutes later all three returned results — Argus found real gaps (7 of 11 entities missing `read: allow` in opencode config), Janus declared the team clean, Aegis issued a DRIFTING verdict. The Argus→Salus repair loop then closed without human direction.

5. **Git log — not parsed agent output — is the canonical verification step.** VESTA-SPEC-054 §3.1-3.2 explains why: "Agent output is conversational." An entity may describe intermediate steps that do not represent the committed result. Parsing this is fragile. The entity's commits are ground truth. Standard verification is `git -C /home/koad/.<entity>/ log --oneline -5`.

6. **The cross-entity commit policy was born from an actual governance incident.** On 2026-04-05, Janus flagged Sibyl's commits to koad/vulcan as a potential unauthorized cross-entity action (LOGS/2026-04-05-day6.md, issue #52). Ruling: the commits were authorized, because Juno had issued the directive and Sibyl referenced the relevant issue. Policy formalized: all future cross-entity commits must include a reference to the directing issue. This is in GOVERNANCE.md (note dated 2026-04-05).

## The GitHub Issues Question

Why GitHub Issues rather than direct calls between entities?

The answer has three layers:

**Persistence.** Session-scoped Agent tool invocations are ephemeral — the only record is what the entity committed. When work spans sessions, spans machines, or involves a human bottleneck (koad), there must be a shared artifact that survives the session. Issues survive. They timestamp. They attribute. They remain open as reminders.

**Attribution.** OPERATIONS.md documents the full sovereignty stack (Layer 1: Hardware through Layer 7: Communication). The stated principle is: "Every layer is owned. Every layer is on disk. Every layer is auditable." Direct API calls between entities have no auditable layer. GitHub Issues are public records tied to authenticated identities.

**Async coordination.** OPERATIONS.md's comparison table (section "What This Replaces") names the key shift: "Juno delegates by spawning" → "Juno delegates by filing issues." "Entity work is ephemeral" → "Entity work is tracked, auditable." The issues-as-protocol pattern enables entities to work on their own schedule, on different machines, without requiring synchronized sessions.
