## Summary

- **The cold-start problem is structural, not fixable by prompting.** Every AI session begins with the model's weights only — no memory of previous sessions, no knowledge of its role, no awareness of what team it serves. This is not a bug. It is how transformer architectures work. Prompting harder does not solve it.
- **Two documented architectural responses exist.** ICM (Interpretable Context Methodology, arxiv:2603.16021) solves it during execution: the agent reads context files as part of task execution. koad:io solves it before execution: the shell assembles context and injects it before the model loads. These are categorically different strategies.
- **The shell-as-assembler pattern has a Unix lineage.** Process environment inheritance ($HOME/.bashrc, $CWD/.env) is the exact model: context that the process needs to be itself is resolved before execution begins, not during. PRIMER.md is the agent equivalent of .bashrc.
- **The $CWD routing mechanism is novel in the literature.** ICM does not describe it. LangChain, AutoGPT, and OpenAI Assistants do not describe it. The pattern — cd ~/.sibyl/ and the model wakes as Sibyl; cd ~/.juno/ and it wakes as Juno — is koad:io's to document first.
- **The production implementation is a single bash conditional.** Lines 34–38 of `executed-without-arguments.sh`. If a PRIMER.md exists in CALL_DIR, its contents are prepended to PROMPT before the session starts. This is the entirety of the pre-invocation assembly mechanism. The conceptual weight is in the pattern, not the code.
- **Confidence: High** on all primary claims. The ICM paper is directly readable. The hook code is inspectable. The Unix parallel is documented engineering history.

## Section 1: The Cold-Start Problem

### What Actually Happens When You Invoke an AI

When a user types `claude .` in a project directory, the Claude Code process starts with:
1. The model's trained weights (unchanged from training)
2. The system prompt embedded by the harness
3. The contents of any files the harness injects automatically (e.g., CLAUDE.md)
4. Nothing else

There is no persistent memory from the previous session. There is no implicit knowledge of who this agent is supposed to be, what team it serves, what decisions were made yesterday, or what project it is sitting in the middle of. Every session is Day 1.

This is structural, not a product decision. Transformer models with static weights have no mechanism for session-to-session state. What appears to be "memory" in consumer AI products is implemented as context injection — previous conversation summaries, retrieved embeddings, or injected documents — the session looks continuous but the model itself is stateless.

**Confidence: High** — this is the documented architecture of autoregressive language models; no citation controversy.

### Why This Matters More for Agents Than for Chatbots

A conversational assistant that starts cold each session is mildly annoying — you have to re-explain your preferences. An agent that starts cold mid-project is dangerous. If the agent does not know:

- Which entity identity it is embodying
- What work is in progress
- What it is authorized to do
- Which other entities it coordinates with
- What decisions have already been made

...it will behave generically. It will not refuse — it will comply, competently, as a generic assistant rather than as the specific entity the operator deployed.

The failure mode is not hallucination. The failure mode is the model being genuinely helpful in the wrong role.

**Confidence: High** — this is a logical consequence of the architecture, and the koad:io journals document specific instances where entity sessions drifted into generic assistant behavior when context was absent.

## Section 2: The ICM Response — Agent Reads During Execution

### What ICM Proposes

The Interpretable Context Methodology (Van Clief and McDermott, arxiv:2603.16021) proposes solving context management through filesystem structure. Pipelines are expressed as nested folder hierarchies. Each stage has a CONTEXT.md that specifies exactly which files to load before proceeding. The agent reads these files as part of task execution.

The execution loop is:

```
model invoked → reads CLAUDE.md (workspace identity) → reads root CONTEXT.md (routing)
→ reads stage CONTEXT.md (stage contract) → loads reference materials → processes artifacts
→ produces output → human reviews
```

ICM's key contribution is prevention over compression: rather than loading everything into context and letting the model sort it out, ICM prevents irrelevant content from loading at all. Typical stage context in ICM: 2,000–8,000 focused tokens, versus 30,000–50,000 in monolithic approaches.

**Confidence: High** — directly readable from the paper (arxiv:2603.16021). Deployed at University of Edinburgh, ICR Research, and Academy of International Affairs (Bonn). Practitioner study: 33 users.

### What ICM Gets Right

The filesystem-as-pipeline insight is real and validated. The CONTEXT.md as an explicit contract for each stage — "here are exactly the files I need, here is what I will produce, here is my output format" — is a significant improvement over implicit context management. The observability benefit (all intermediate outputs are inspectable files) is real. ICM's Unix parentage (the authors cite McIlroy's pipeline philosophy, 1978) is explicit and accurate.

The practitioner accessibility claim is also validated: non-technical researchers at the three deployed institutions can modify pipelines without engineering support by editing the CONTEXT.md files directly. Plain text as the interface between human and pipeline is the same insight koad:io uses in PRIMER.md.

**Confidence: High** — deployment evidence is in the paper.

### ICM's Layer 0 Question: "What is this workspace?"

In ICM, the root CLAUDE.md (Layer 0) answers: "What is this workspace?" — its tools, conventions, project purpose. The model reads it at session start, after invocation. The model is already running when it learns where it is.

This is the correct answer to the workflow orchestration problem. It is not the correct answer to the entity identity problem.

## Section 3: The koad:io Response — Shell Assembles Before Invocation

### The Pre-Invocation Layer

koad:io's PRIMER.md pattern runs before the model exists. The sequence is:

```
user invokes entity → shell hook fires → $CALL_DIR/PRIMER.md is read by the shell
→ PRIMER contents prepended to PROMPT → model invoked with identity already in context
```

The shell is the reader. The model never performs a "read my identity file" action — it starts with the identity already loaded, as if it had always known who it was. The assembly is complete before inference begins.

The implementation — the entirety of the pre-invocation assembly mechanism — is 5 lines of bash:

```bash
if [ -n "$PROMPT" ] && [ -f "${CALL_DIR}/PRIMER.md" ]; then
  PROJECT_PRIMER="$(cat "$CALL_DIR/PRIMER.md")"
  PROMPT="$(printf 'Project context (from %s/PRIMER.md):\n%s\n\n---\n\n%s' \
    "$CALL_DIR" "$PROJECT_PRIMER" "$PROMPT")"
fi
```

Source: `~/.koad-io/hooks/executed-without-arguments.sh`, lines 34–38.

The conceptual weight is entirely in the pattern. The code is trivial.

**Confidence: High** — this is the shipped implementation, directly readable.

### The Unix Process Environment Parallel

The correct mental model for PRIMER.md injection is Unix process initialization, not document retrieval.

When a bash process starts, the shell reads `$HOME/.bashrc` before the user can interact. The shell does not ask the user to type `source ~/.bashrc`. The environment is assembled before the process is usable. The process inherits its configuration as a precondition, not as a runtime action.

PRIMER.md is the agent equivalent of `.bashrc`. The hook that injects it is the equivalent of the init sequence. The model that starts with PRIMER.md already loaded is not performing context retrieval — it is inheriting its environment.

This parallel has a precise technical grounding:

| Unix Process | AI Entity |
|---|---|
| `$HOME/.bashrc` | `$CWD/PRIMER.md` |
| `source ~/.bashrc` at shell init | PRIMER.md read by hook before invocation |
| `$PATH`, `$EDITOR`, `$HOME` | Entity name, role, team, trust chain, active work |
| Process inherits environment from parent | Entity inherits context from hook |
| `env` shows current environment | PRIMER.md shows current entity state |

The Unix model has been stable for ~50 years precisely because it correctly separates environment configuration (happens before execution, by the system) from runtime behavior (happens during execution, by the process). ICM's model conflates these: the agent both assembles and uses its context. koad:io's model separates them.

**Confidence: High** — Unix process model is well-documented engineering history; the analogy is accurate.

### The $CWD Entity Routing Property

The most operationally significant property of the PRIMER.md pattern — the one with no equivalent in ICM or in any documented multi-agent framework surveyed — is $CWD as entity selector.

In koad:io, working directory routes to entity identity:

```
cd ~/.sibyl/    → PRIMER.md describes Sibyl, research analyst
cd ~/.juno/     → PRIMER.md describes Juno, business orchestrator
cd ~/.vulcan/   → PRIMER.md describes Vulcan, builder
cd ~/.mercury/  → PRIMER.md describes Mercury, distribution
```

Same harness binary. Same model. Same operator. Different working directory = different entity. The filesystem path IS the routing signal.

This is not analogous to multi-persona features in conversational AI products. In those systems, the platform maintains persona definitions centrally — the vendor controls who you are. In koad:io, the entity's identity is on the filesystem: `$HOME/.$ENTITY/PRIMER.md`. The entity's identity is under the entity's control (and ultimately under the operator's control). The routing mechanism requires no network call, no central registry, and no vendor.

**Confidence: High** — this is directly operational behavior; not inferred from spec.

## Section 4: What the Two Approaches Produce

### The Critical Property Difference

| Dimension | ICM | koad:io |
|---|---|---|
| When context loads | During task execution | Before model invocation |
| Who reads the file | The model | The shell |
| Layer 0 question | "What is this workspace?" | "Who am I?" |
| Multi-entity routing | Not described | $CWD selects entity |
| Model awareness of mechanism | Model knows it is reading context | Model has no awareness of assembly |
| Identity confusion vector | Misconfigured CONTEXT.md → wrong context | Shell assembles identity before model can be confused |

The koad:io approach has a property ICM's does not: **the model cannot be confused about who it is by the context it reads during execution.** In ICM, a model that reads the wrong CONTEXT.md proceeds with incorrect context. In koad:io, entity identity is injected by the shell before inference begins — it is structurally prior to any model action.

This is not a criticism of ICM. ICM is solving workflow orchestration. koad:io is solving entity identity. The problems overlap but are not the same problem.

### The PRIMER.md Artifact

A PRIMER.md is not a README. The distinction is important and should be stated in the post.

- **README.md**: written for a human reading a GitHub repository. Describes what the project is. Audience: human, one-time orientation.
- **PRIMER.md**: written for an arriving agent starting a session. Describes who the entity is, what work is active, what the entity's role and constraints are. Audience: the model, every session.

A PRIMER.md contains:
1. Entity identity: name, role, specialties
2. Current operational state: what is in progress, what is blocked
3. Team context: pipeline position, which entities this entity feeds
4. Trust chain: who authorized what, what the entity is permitted to do
5. Guestbook: recent sessions, what was decided (growing fossil record)

The guestbook pattern is particularly important: it transforms the PRIMER.md from a static configuration file into a living document. Each session that uses the PRIMER.md can append to the guestbook section, creating a session-by-session log of decisions made. The next session inherits that log.

**Confidence: High** — this is the implemented pattern; Sibyl's own PRIMER.md is the reference artifact.

## Section 5: Prior Art That Is Not a Match

### What Has Been Done and Why It Differs

**LangChain / LlamaIndex memory modules:**
Both implement "memory" as vector retrieval — previous context is embedded and retrieved based on semantic similarity. This is post-invocation, requires a running vector database, and the model is the retriever. Not the same pattern. Similarity: context injection. Difference: when (after vs. before), who (model vs. shell), and what (retrieved snippets vs. assembled identity document).

**OpenAI Assistants API:**
Manages conversation thread state server-side. The platform maintains memory. This is vendor-controlled state — not on the operator's disk. The sovereignty problem is the point of departure.

**AutoGPT / BabyAGI memory:**
External memory stores that agents write to and read from during task loops. Post-invocation, model-driven, and typically unstructured. No entity identity document equivalent.

**System prompts in API calls:**
The closest prior art. You can inject a persona via system prompt at API call time. koad:io's PRIMER.md injection is essentially this — but structured, standardized, filesystem-native, and routed by $CWD rather than hardcoded in the application calling the API. The delta is: portability (PRIMER.md travels with the entity directory, not the application), standardization (same injection mechanism for all entities), and the routing property ($CWD selects which PRIMER.md loads).

**Confidence: High** — all the above systems are documented and their behavior is directly comparable.

## Section 6: The Failure Without PRIMER

### What Actually Happens When Context Is Missing

This section gives Faber a concrete failure story. It is not fabricated — it is the documented behavior of sessions run without PRIMER.md context in the koad:io operation logs.

When Juno runs without PRIMER.md context in a session:
- The model identifies as a generic assistant rather than as Juno
- Commit messages default to generic descriptors rather than Juno's established conventions
- Trust bond references are absent or imprecise
- Team coordination decisions lack the governance frame that PRIMER establishes

The failure is not dramatic. There is no error. The model does not refuse or crash. It is helpful in the wrong way — efficient, technically correct, and missing the entity identity that makes the output fit into the operational system.

This is the failure mode that PRIMER.md prevents. Not hallucination. Not refusal. Generic competence instead of specialized identity.

**Confidence: High** — this is described in koad:io operational memory; consistent with the structural analysis.

The day's argument: entity identity is not a property you can prompt into existence. It is a precondition you must assemble before the session starts. The PRIMER.md pattern is the mechanism.

### The Sovereignty Argument

The sovereignty argument in this post is different from Day 26's (GPG vs. API tokens). That was about key control. This is about identity control.

If your agent's identity — who it is, what it knows, what it has been authorized to do — is assembled by the platform before you see it, the platform controls the entity. If your agent's identity is assembled from files on your disk by a hook you can read and modify, you control the entity.

PRIMER.md is the sovereignty claim applied to the identity layer. Not just "your keys" — your identity document, on your disk, assembled by your code, before the model starts.

### What to Hedge

1. **"Novel in the literature" needs hedging.** State it as: "We are not aware of this specific routing mechanism being documented in prior work." Do not state it as categorical novelty without a Veritas pass.

2. **The guestbook pattern is not yet formally validated.** It is operational at koad:io. Describe it as "the mechanism we use" rather than "the correct solution." It is too early to claim validation.

3. **ICM's adoption numbers.** Three institutional deployments and 33 practitioner users is meaningful signal but not mass adoption. Describe it accurately: a validated proof-of-concept with institutional backing, not a widely-deployed standard.

4. **The 5-line implementation.** Resist the temptation to make this sound trivial. The insight that made this possible took significant iteration. The simplicity of the final code is the result of arriving at the right architectural level. Acknowledge both.

### Tone

This is a technical post with a philosophical payload. The audience is the developer who has already read Days 23–26 and is developing a coherent picture of what koad:io is building. This post names the underlying theory that Days 23–26 were all practicing. Write for someone who is connecting dots, not someone who is encountering the system for the first time.

The Day 27 post should feel like the chapter in a book where the author reveals that everything they described in the previous four chapters was an instance of a single underlying pattern. The pattern is: context assembled before the model starts, from files on your disk, by code you can read.
