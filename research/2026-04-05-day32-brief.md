## What Makes the Vulcan Exception Architecturally Significant

### The bond type difference

Every team entity has a trust bond from Juno. The bond types split into two categories:

| Entity | Bond Type | What It Grants |
|--------|-----------|----------------|
| Vulcan | `authorized-builder` | "Accept build assignments filed as GitHub Issues... create and commit code... gestate new entity flavor repos" |
| Mercury | `peer` | "File GitHub Issues on each other's repos to request work... coordinate directly" |
| Sibyl | `peer` | Same as Mercury |
| Veritas | `peer` | Same as Mercury |
| Muse | `peer` | Same as Mercury |

The `authorized-builder` bond is not a stronger version of `peer` — it is a different relationship type. A peer bond grants bilateral coordination rights with no subordination. An authorized-builder bond grants unidirectional build authority: Vulcan builds what Juno files. Vulcan cannot initiate build projects without a Juno-filed issue. Vulcan cannot represent Juno in external communications.

This distinction is in the bond itself: "Vulcan is NOT authorized to: initiate build projects without a Juno-filed GitHub Issue" (`juno-to-vulcan.md`, Authorized Actions, NOT authorized list).

### The invocation consequence

Because Vulcan's relationship to Juno is `authorized-builder` rather than `peer`, the invocation protocol is different by design, not by accident.

VESTA-SPEC-054 §1.3: "Vulcan is never invoked via the Agent tool. He builds on wonderland, paired with Astro. Work for Vulcan goes as a GitHub Issue on `koad/vulcan`."

VESTA-SPEC-053 §6 specifies the constraint table directly:

| Constraint | Value |
|-----------|-------|
| Permitted build host | wonderland |
| Permitted pairing | with Astro |
| Invocation method | GitHub Issues (not Agent tool) |
| Direct local invocation | Never |

The Agent tool invocation pattern — which works for every other entity — does not apply to Vulcan. If Juno needs Vulcan to build something, Juno files an issue on `koad/vulcan`. Vulcan picks it up in his next session on wonderland. He does not receive a brief, run, and return results to Juno in the same session.

### Why wonderland and Astro

VESTA-SPEC-053 §6 explains the operational reason: "Wonderland has the uncommitted Alice work, the active build environment, and the specific context that Vulcan and Astro have co-developed. Migrating Vulcan to thinker or fourty4 and having him build there would produce worse outcomes."

This is a human-in-the-loop choice embedded in the architecture. The spec is explicit: "This is not a technical limitation — it is an operational decision made by koad." The Vulcan exception is not a bug or an incomplete feature. It is the design choice that koad made about where Vulcan does his best work — and that choice is formalized in the trust bond and the portability spec.

### What "code review when the reviewer is an AI" means here

Juno is the reviewer of Vulcan's work. The workflow protocol in `juno-to-vulcan.md` is explicit:

1. Assignment: Juno files GitHub Issue on `koad/vulcan` with product spec
2. Acknowledgment: Vulcan comments on issue to confirm receipt and scope
3. Build: Vulcan builds, commits to assigned repo, updates issue with progress
4. Delivery: Vulcan comments with completion summary and links
5. Verification: **Juno reviews, closes issue if accepted; comments if changes needed**

Juno reviews Vulcan's commits — an AI reviewing an AI's work. The governance mechanism that makes this meaningful is the same one that makes all entity work auditable: every action is committed to a repo with a git identity, referenced to an issue, and traceable through the trust bond chain. Juno's verification is not informal approval — it is the official closure event in a cryptographically attributed workflow.

The trust chain makes clear why this is safe: `koad → Juno (authorized-agent) → Vulcan (authorized-builder)`. Vulcan's authorized scope is bounded by Juno's scope, which is bounded by koad's authorization. Juno cannot authorize Vulcan to do things Juno herself is not authorized to do. If koad revokes the koad→Juno bond, all downstream bonds including juno→Vulcan are suspended.
