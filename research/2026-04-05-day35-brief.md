# Day 35 Research Brief: The Parallel Session Pattern

## The story

Days 31–34 have built a specific picture of how the koad:io team operates: the orchestration loop (31), the Vulcan exception (32), the Operations Board as visibility layer (33), the review chain as error surface (34). All of these describe the *rules* of the system — the protocol, the bond types, the label taxonomy, the review chain. Day 35 should show the system running at full stretch, in real time.

The Day 6 log (`~/.juno/LOGS/2026-04-05-day6-entity-portability-session.md`) documents the most concrete multi-entity parallel session on record: 9 agents running simultaneously across a single session, with the orchestrator (Juno) continuing to assign work as each agent reported back. The session ran through four distinct phases — each marked by a context window compression event — and produced a verifiable artifact record across nine entities.

This is not a hypothetical walkthrough. The git logs, commit hashes, and output files from that session are on disk. Faber can cite specific commits (`015050f`, `35249fd`, `d8eb1ee`) and specific agent outputs. The session documented itself as it ran.

The strongest story for Day 35: take the reader inside one parallel session from first launch to final commit. Show how context compression is handled, how git-log verification works, how the orchestrator decides what to launch next, and what the failure mode looks like (the Chiron hook routing to fourty4 with expired auth — found and fixed in the same session). This is the arcs of Days 31–34 running in parallel, live.

## Why this angle over the alternatives

**Budget angle:** Copia and the April CAD 1,000 ceiling is a real artifact, but the compelling claim (financial constraint shapes entity decisions) requires showing a decision that was actually changed by budget — and the April budget's most dramatic event (the Day 29 cost correction from $24/month to ~$105/month) was already covered in the Day 34 post. Retreading it weakens both posts.

**Daemon angle:** The daemon architecture (`~/.koad-io/daemon`, `passenger.json`, the worker system) is architecturally interesting but partly aspirational — dotsh SSH is on the blocked list (`koad/juno#56`), and the daemon's first run is documented as "in progress" as of 2026-04-03. A post about the daemon has to explain what it doesn't yet do. That is a harder narrative to make concrete.

**Context window angle:** This is real and the failure mode is verifiable (four compression events in the Day 6 session log), but the context window management story is best told *as part of* the parallel session story, not as its own post. The compression events are what reveal what survives (PRIMER.md, memories/, the standing directive) and what gets reconstructed (the specific agent statuses, the in-progress list). The mechanism only becomes interesting in the context of the session that demonstrated it.

**Parallel session angle:** The Day 6 log is an exceptionally rich artifact for this post. It documents: launch order for 9 agents, what each produced, the context compression boundary (explicit "Session resumed from compressed summary" notes mark each new phase), a live failure catch (Chiron hook), the standing directive persistence across compressions ("keep going until all done or blocked by me or Vulcan"), and the agent-notification-then-verify loop. Every claim Faber needs to make has a specific file, commit, or session log line to cite.

## What to avoid

**Don't frame this as impressive scale.** The interesting thing is not that 9 agents ran in parallel — it is that the system for doing so is reproducible, verifiable, and leaves a clear artifact trail. The post should feel like a technical walkthrough, not a capabilities showcase.

**Don't elide the failure.** The Chiron hook failure is the most interesting single event in the session log. It should get prominent treatment. A system that catches and fixes its own routing errors during a compression event is a more credible system than one that describes how everything went smoothly.

**Don't conflate context compression with data loss.** The session log explicitly reconstructs from committed state each time. The post should be precise: compression doesn't lose work (it's committed), it loses conversational context (which session-scoped decisions were being weighed). The PRIMER/memories/CLAUDE.md layer handles the former; git log handles verification; the session log captures the narrative for later reconstruction.

**Don't treat the Day 6 session as a past event.** The form of the post should be present-tense description of a documented pattern, not retrospective. "This is how it works. Here is the record of it working."

## Structure suggestion

1. **The standing directive** — Open with the sentence: "keep going until all done or blocked by me or Vulcan." Not koad's message — it survived four context window compressions and appeared verbatim each time the session reconstructed. That is the most compressed summary of how the system runs at full stretch.

2. **What a parallel session looks like** — Walk through the Day 6 first phase: 9 agents launched, each assigned to independent work, all with `run_in_background: true`. Not abstract — Chiron writing curriculum atoms, Vesta writing specs, Faber writing posts, Argus running the health check.

3. **Context compression as the session structure** — Four phases, four compressions. What each compression reset (conversational state) and what it didn't (committed work, standing directive, blocked list). The compression is not a failure mode — it is the cadence. Each phase starts with a git-log reconstruction and proceeds.

4. **The failure the compression surfaced** — Chiron's hook routing to fourty4. How the compression event was the mechanism that surfaced the error. The fix: two memory records and an updated architecture file. Committed, resolved.

5. **What the session produced** — The per-entity volume table. Not to impress — to show that the git log is the complete record. Every one of those atoms, specs, posts, and reviews has a commit hash. The session is auditable to any line.

6. **What you can't see** — Acknowledge the gap: the Operations Board and session log document the work, but the conversational decisions — why Sibyl was asked for the ICM synthesis brief while Faber was mid-draft on Day 26 — are not committed anywhere. The orchestrator's judgment is the one thing the fossil record doesn't preserve.
